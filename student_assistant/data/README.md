# Knowledge Base Builder (SING Project)

Система автоматизированной сборки, очистки и валидации единой базы знаний (FAQ) из разнородных источников. Проект подготавливает структурированный JSON для работы Telegram-бота.

---

## 🏗 Архитектура пайплайна

Обработка данных разделена на четыре логических этапа:

1.  **📥**Конвертация (`convert_*.py`)**: Приведение сырых данных (лекции, выгрузки из чатов) к единому стандарту FAQ.
2.  **🔄**Слияние**: Объединение всех веток (Manual + Lectures + Chats) в один массив.
3.  **🧹**Дедупликация (`dedup.py`)**: Очистка базы от похожих вопросов. Использует коэффициент Жаккара (сравнение множеств лемм) и проверку коэффициента длины строк для защиты уникального контента.
4.  **✅**Валидация (`validate.py`)**: Проверка финального файла на соответствие схеме, уникальность ID и корректность категорий.
5.  **💾 ЭКСПОРТ (Load)**: Финальный, «чистый» массив сохраняется в `faq.json` для Telegram-бота, а статистика сборки записывается в `report.txt`.

---

## 🚀 Быстрый запуск

### 1. Требования
Для работы скриптов необходим Python 3.8+ и следующие библиотеки:

	pip install nltk pymorphy3

## 2. Подготовка данных
Поместите исходные файлы в директорию sources/ (настраивается в build_faq.py):

`faq_manual.json` — ручные, эталонные записи.

`lectures.json` и `lecture_faq.json` — данные и вопросы по лекциям.

`student_questions.json` — сырые вопросы из чатов.

## 3. Запуск сборки
	python build_faq.py
Результат: файл faq.json (основная база) и report.txt (статистика сборки).

## ⚙️ Описание модулей

### `build_faq.py`
Главный скрипт-оркестратор, который управляет всем жизненным циклом сборки базы данных. Он последовательно запускает конвертацию, слияние, очистку и проверку данных.

**Глобальные настройки:**
* `FILES`: Словарь с путями к исходным файлам (`sources/`), результирующему JSON, логам ошибок и текстовому отчету.
* `DEDUP_THRESHOLD`: Установлен на **95%**. Это высокий порог сходства, необходимый для минимизации ложных удалений (лучше оставить похожий вопрос, чем удалить уникальный).

---

### `dedup.py`
Модуль «умной» дедупликации, работающий в два этапа (точное совпадение строк и нечеткий поиск по коэффициенту Жаккара).

**Логика работы:**
* **Слияние данных:** Если два вопроса признаны похожими, скрипт не просто удаляет один из них, а пытается сохранить максимум информации. Приоритет отдается записи, где поле `answer` заполнено (актуально при объединении данных из чатов с ручными записями).
* **Механизм защиты:** Введена проверка длины строк. Если текст одного вопроса короче другого более чем на **40%**, они **не** считаются дубликатами, даже если ключевые слова совпадают. Это защищает короткие уточняющие вопросы от поглощения более развернутыми.

---

### `validate.py`
Модуль финального контроля качества, который гарантирует, что Telegram-бот сможет корректно прочитать и обработать файл.

**Критерии проверки:**
* **Валидация категорий:** Проверка на соответствие строгому списку: *навигация, экзамены, преподаватели, учебный_процесс, ит_системы, бюрократия, общежитие, лекции*.
* **Целостность данных:** Проверка наличия всех обязательных полей (`id`, `question`, `answer`, `keywords`, `category`, `source`) и их типов.
* **Система идентификаторов (ID):** Контроль уникальности и вхождения в заданные диапазоны:
    * `1 – 999`: Ручные записи (**Manual**)
    * `1000-1999`: Данные из лекций (**Lectures**)
    * `2000+`: Вопросы из студенческих сообществ (**Chats**)

# 📊 Формат записи (FAQ Entry)

Все данные в итоговом файле `faq.json` строго типизированы. Это гарантирует, что поисковый алгоритм Telegram-бота будет работать без ошибок.

| Поле | Тип | Обяз. | Описание |
| :--- | :--- | :---: | :--- |
| `id` | `int` | Да | Уникальный ID. **1–999**: бытовой FAQ, **1000–1999**: лекции, **2000+**: из чатов |
| `question` | `str` | Да | Полная формулировка вопроса |
| `keywords` | `list[str]` | Да | Ключевые слова для поиска (строчные леммы, без знаков препинания) |
| `answer` | `str` | Да | Текст ответа |
| `category` | `str` | Да | Одна из: `навигация`, `экзамены`, `преподаватели`, `учебный_процесс`, `ит_системы`, `бюрократия`, `общежитие`, `лекции` |
| `source` | `str` | Да | Откуда запись: `manual`, `chat_экзамены_мирэа`, `chat_поступление_иит_2024`, `chat_подслушано_мирэа`, `lecture_<id>` |

## 📝 Логирование и отчетность

После завершения цикла сборки система автоматически генерирует файл `report.txt`. Этот отчет служит инструментом контроля качества и позволяет быстро оценить состояние базы знаний.

### Состав отчета:
* **Статистика по источникам:** количество успешно импортированных записей из каждого файла с указанием выделенных диапазонов ID.
* **Результаты очистки:** точное количество удаленных дубликатов на этапе дедупликации.
* **Анализ категорий:** распределение вопросов по тематикам (позволяет отследить перекосы в базе данных).
* **Статус валидации:** финальный вердикт системы и детальный список ошибок (если они были обнаружены).
